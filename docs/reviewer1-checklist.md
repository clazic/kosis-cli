# 평가자1 체크리스트 (CLI/help 엄격 평가)

담당 범위:
- `cmd/data.go`
- `cmd/meta.go`
- `cmd/explain.go`
- `cmd/root.go`
- `cmd/indicator.go` (`indicator --help`, `ind data --help` 포함)

기준 문서:
- 설계서: `../../docs/superpowers/specs/2026-03-31-kosis-cli-design.md`
- 사용 가이드: `../../skills/kosis-cli/SKILL.md`

평가 원칙:
- help 문구와 실제 파싱/실행 결과가 다르면 즉시 감점한다.
- 문서화된 예제가 실행 불가능하면 치명 결함으로 본다.
- 100점 미만은 무조건 재작업이다.

## 1. 평가 절차

검증 명령:
- [ ] `go run . --help`
- [ ] `go run . data --help`
- [ ] `go run . meta --help`
- [ ] `go run . explain --help`
- [ ] `go run . indicator --help`
- [ ] `go run . ind data --help`
- [ ] `go run . data 101 DT_TEST -c1 11 -i T10 -p M` (파싱/오류 형태 확인)
- [ ] `go run . meta 101 DT_1IN1502 -f json` (포맷 반영 확인)
- [ ] `go run . explain` (인자 생략 정책 확인)

## 2. 배점표 (총 100점)

### A. 공통 help 형식 준수 (20점)
- [ ] 한줄 설명, 상세 설명, 사용법, 파라미터, 플래그, 예제, 다음 단계가 설계서 형식을 따른다. (10점)
- [ ] 별칭 표기와 실제 alias가 일치한다. (5점)
- [ ] 구현되지 않은 기능을 완료된 것처럼 안내하지 않는다. (5점)

### B. `data` 계약 일치성 (30점)
- [ ] help 예제가 실제 파싱 가능하다. (10점)
- [ ] 플래그 표기(`-c1`/`--class1` 등)와 실제 동작이 충돌하지 않는다. (8점)
- [ ] `--latest` 타입/검증/오류 메시지가 합리적이다. (4점)
- [ ] interactive 이후 재사용 원라인 명령이 실제 문법과 맞다. (4점)
- [ ] 에러 메시지가 다음 행동(`meta` 확인 등)을 안내한다. (4점)

### C. `meta` 계약 일치성 (20점)
- [ ] `-f json` 요청 시 summary/typed 모드 모두 포맷 정책이 일관된다. (10점)
- [ ] 출력 문자열 인코딩/깨짐 문제가 없다. (5점)
- [ ] help 설명과 실제 출력 구조가 어긋나지 않는다. (5점)

### D. `explain` 대화형/가이드 정책 (15점)
- [ ] 인자 생략 시 설계서 원칙(대화형) 또는 명확한 제한 안내 중 하나를 충족한다. (8점)
- [ ] help 예제와 실제 실행 경로가 일치한다. (7점)

### E. root/indicator 보조 기준 (15점)
- [ ] root help가 현재 구현 상태와 정합적이다. (8점)
- [ ] indicator 부모 help에 하위 명령 사용 흐름이 충분히 표현된다. (7점)

## 3. 치명 결함 (즉시 FAIL)

아래 항목 중 하나라도 발생하면 해당 파일 점수는 최대 79점:
- [ ] help 예제 명령이 파싱 단계에서 실패
- [ ] 문서화된 플래그가 실제로 존재하지 않음
- [ ] 포맷 플래그(`-f json`)가 무시됨
- [ ] 인코딩 깨짐으로 의미 전달 실패
